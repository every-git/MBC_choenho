## 🌿 GitHub 사용법 (처음 쓰는 사람을 위한 상세 가이드!)

> ⚠️ **중요**: GitHub를 처음 사용하는 분들을 위해 단계별로 자세히 설명합니다!

### 브랜치 구조
```
main (모든 사람의 작업이 합쳐지는 곳)
  ├── feature/정용준 (리더)
  ├── feature/선하영
  ├── feature/오창준
  └── feature/최명헌
```

### 📚 GitHub 처음 시작하기 (단계별 가이드)

#### Step 1: GitHub 계정 만들기 (처음 한 번만)
1. https://github.com 접속
2. "Sign up" 클릭
3. 이메일, 비밀번호 입력
4. 계정 생성 완료

#### Step 2: Git 설치 확인 (처음 한 번만)
```bash
# 터미널(명령 프롬프트)에서 확인
git --version

# 설치 안 되어 있으면 다운로드
# Windows: https://git-scm.com/download/win
# Mac: 터미널에서 "git" 입력하면 자동 설치 안내
```

#### Step 3: Git 기본 설정 (처음 한 번만)
```bash
# 자신의 이름과 이메일 설정
git config --global user.name "자신의 이름"
git config --global user.email "자신의 이메일"

# 예시
git config --global user.name "정용준"
git config --global user.email "yongjun@example.com"
```

#### Step 4: 저장소 받아오기 (모든 팀원)
```bash
# 리더가 만든 저장소 URL 복사
# 예: https://github.com/정용준/shopping-project.git

# 터미널에서 실행
git clone https://github.com/정용준/shopping-project.git
cd shopping-project

# 성공하면 프로젝트 폴더가 생성됨!
```

#### Step 5: 각자 작업 공간 만들기
```bash
# 정용준 (리더)
git checkout -b feature/정용준

# 선하영
git checkout -b feature/선하영

# 오창준
git checkout -b feature/오창준

# 최명헌
git checkout -b feature/최명헌
```

**확인 방법**: `git branch` 입력하면 현재 브랜치에 * 표시가 있음

### 📝 매일 작업하는 방법 (매우 중요!)

#### 🌅 아침에 할 일 (매일 반복!)
```bash
# 1단계: 현재 어느 브랜치에 있는지 확인
git branch
# * 표시가 있는 브랜치가 현재 위치

# 2단계: main 브랜치로 이동
git checkout main

# 3단계: GitHub에서 최신 코드 가져오기
git pull origin main
# 이렇게 하면 다른 사람이 올린 최신 코드를 받아옴!

# 4단계: 내 작업 공간으로 이동
git checkout feature/자신이름
# 예: git checkout feature/선하영

# 5단계: main의 최신 변경사항을 내 작업 공간에 가져오기
git merge main
# 이렇게 하면 다른 사람이 만든 기능을 내 작업에도 반영됨!
```

**왜 이렇게 하나요?**
- 다른 사람이 만든 기능을 내가 사용할 수 있게 하기 위해
- 충돌을 미리 방지하기 위해

#### 💻 작업하고 저장하기
```bash
# 1단계: 코드 작성 및 테스트
# (IDE에서 코드 작성하고 실행해서 확인)

# 2단계: 변경된 파일 확인
git status
# 빨간색으로 표시되는 파일들이 변경된 파일

# 3단계: 변경사항 저장 준비
git add .
# . 은 "모든 변경사항"을 의미함
# 특정 파일만 저장하려면: git add 파일명

# 4단계: 저장 (무엇을 했는지 간단히 적기)
git commit -m "로그인 기능 추가"
# -m 뒤에 메시지를 적으면 됨

# 5단계: GitHub에 올리기
git push origin feature/자신이름
# 예: git push origin feature/선하영

# 성공하면 "Everything up-to-date" 또는 "pushed" 메시지가 나옴!
```

#### 🔗 GitHub에서 Pull Request 만들기 (상세 가이드)

**Pull Request란?**
- 내가 만든 코드를 main에 합치고 싶다고 요청하는 것
- 팀원들이 확인하고 승인해야 합쳐짐

**만드는 방법:**
1. GitHub 웹사이트 접속 (https://github.com)
2. 저장소 페이지로 이동
3. 상단에 "Compare & pull request" 버튼이 보임 (파란색)
4. 클릭하면 Pull Request 작성 페이지가 열림
5. 제목과 설명 작성:
   - 제목: "로그인 기능 추가"
   - 설명: "web-study-09에서 로그인 코드를 복사했습니다"
6. "Create pull request" 버튼 클릭
7. 완료! 이제 리더(정용준)가 확인하고 승인하면 main에 합쳐짐

**리더가 할 일:**
1. Pull Request 목록에서 확인
2. 코드 리뷰 (코드가 잘 작성되었는지 확인)
3. 문제 없으면 "Merge pull request" 버튼 클릭
4. 승인 완료!

**확인 방법:**
- Pull Request가 승인되면 GitHub에서 알림이 옴
- main 브랜치에 내 코드가 합쳐졌는지 확인 가능

## 💡 GitHub 용어 쉽게 이해하기 (초보자 필독!)

### 저장소 (Repository, Repo)
- **쉽게 말하면**: 프로젝트가 저장되는 공간
- **예시**: "shopping-project"라는 이름의 저장소
- **비유**: 네이버 클라우드나 구글 드라이브의 폴더 같은 것
- **왜 필요한가**: 모든 코드를 한 곳에 모아서 관리하기 위해

### 브랜치 (Branch)
- **쉽게 말하면**: 각자 작업하는 독립된 공간
- **예시**: 
  - `main`: 모든 사람의 작업이 합쳐지는 메인 공간
  - `feature/선하영`: 선하영이 작업하는 공간
  - `feature/오창준`: 오창준이 작업하는 공간
- **비유**: 각자 다른 방에서 작업하는 것처럼
- **왜 필요한가**: 서로 방해받지 않고 각자 작업하기 위해

### 커밋 (Commit)
- **쉽게 말하면**: 작업한 내용을 저장하는 것 (저장 버튼 누르기)
- **예시**: "로그인 기능 추가" 라고 적고 저장
- **비유**: 게임에서 세이브 포인트를 만드는 것
- **왜 필요한가**: 
  - 나중에 무엇을 했는지 기억하기 위해
  - 실수했을 때 이전 상태로 돌아갈 수 있게 하기 위해

### Push (푸시)
- **쉽게 말하면**: 내 컴퓨터의 코드를 GitHub에 올리는 것
- **예시**: `git push origin feature/선하영`
- **비유**: 네이버 클라우드에 파일 업로드하는 것
- **왜 필요한가**: 다른 사람들이 내 코드를 볼 수 있게 하기 위해

### Pull (풀)
- **쉽게 말하면**: GitHub의 코드를 내 컴퓨터로 가져오는 것
- **예시**: `git pull origin main`
- **비유**: 네이버 클라우드에서 파일 다운로드하는 것
- **왜 필요한가**: 다른 사람이 만든 최신 코드를 받아오기 위해

### Pull Request (PR)
- **쉽게 말하면**: 내 작업을 main에 합치고 싶다고 요청하는 것
- **예시**: "제가 만든 로그인 기능을 main에 합쳐주세요!"
- **비유**: 선생님께 숙제 제출하고 확인받는 것
- **왜 필요한가**: 
  - 팀원들이 확인하고 승인한 후에만 합치기 위해
  - 코드 품질을 유지하기 위해

### 병합 (Merge)
- **쉽게 말하면**: 다른 사람의 작업을 내 작업에 가져오는 것
- **예시**: 오창준이 만든 상품 등록 기능을 내 작업에 가져오기
- **비유**: 친구가 만든 문서를 내 문서에 복사해 넣는 것
- **왜 필요한가**: 다른 사람이 만든 최신 기능을 사용하기 위해

### 충돌 (Conflict)
- **쉽게 말하면**: 같은 파일을 동시에 수정해서 생기는 문제
- **예시**: 선하영과 오창준이 같은 파일을 동시에 수정함
- **비유**: 두 사람이 같은 문서를 동시에 수정하는 것
- **해결 방법**: 
  1. 팀원과 상의해서 어떤 코드를 사용할지 정하기
  2. 파일에서 `<<<<<<<` 표시가 있는 부분 찾기
  3. 필요한 코드만 남기고 나머지 삭제
  4. 다시 커밋하기

## 🆘 문제 해결 (GitHub 초보자를 위한 상세 가이드)

### 문제 1: "git: command not found" 에러
**원인**: Git이 설치되지 않음
**해결**:
- Windows: https://git-scm.com/download/win 에서 다운로드
- Mac: 터미널에서 `xcode-select --install` 실행
- 설치 후 터미널 재시작

### 문제 2: "Permission denied" 에러
**원인**: GitHub 인증 문제
**해결**:
1. GitHub에서 Personal Access Token 생성
2. 비밀번호 대신 토큰 사용
3. 또는 SSH 키 설정 (고급)

### 문제 3: Git 충돌 발생
**증상**: `git merge` 할 때 "CONFLICT" 메시지가 나옴
**해결**:
```bash
# 1단계: 충돌 파일 확인
git status
# 빨간색으로 표시된 파일이 충돌 파일

# 2단계: 파일 열어서 확인
# 파일 안에 이런 표시가 있음:
# <<<<<<< HEAD
# 내가 작성한 코드
# =======
# 다른 사람이 작성한 코드
# >>>>>>> main

# 3단계: 필요한 코드만 남기고 나머지 삭제
# <<<<<<<, =======, >>>>>>> 표시도 모두 삭제

# 4단계: 수정 완료 후 저장
git add .
git commit -m "충돌 해결"
```

**중요**: 충돌 나면 혼자 해결하려고 하지 말고 리더(정용준)에게 물어보기!

### 문제 4: 잘못 커밋함
**증상**: 커밋 메시지를 잘못 적었거나 잘못된 파일을 커밋함
**해결**:
```bash
# 마지막 커밋 취소 (파일은 그대로 유지)
git reset --soft HEAD~1

# 다시 커밋
git commit -m "올바른 메시지"
```

### 문제 5: 브랜치를 잘못 만듦
**증상**: 브랜치 이름을 잘못 입력함
**해결**:
```bash
# 올바른 브랜치로 이동
git checkout main
git checkout -b feature/올바른이름

# 잘못된 브랜치 삭제
git branch -D feature/잘못된이름
```

### 문제 6: "push rejected" 에러
**원인**: 다른 사람이 먼저 push해서 내 코드가 구버전임
**해결**:
```bash
# 1단계: 최신 코드 가져오기
git pull origin main

# 2단계: 충돌 있으면 해결
# 3단계: 다시 push
git push origin feature/자신이름
```

### 문제 7: 파일을 실수로 삭제함
**해결**:
```bash
# 삭제한 파일 복구
git checkout HEAD -- 파일명

# 또는 모든 변경사항 취소
git checkout .
```

### 문제 8: 어느 브랜치에 있는지 모르겠음
**해결**:
```bash
# 현재 브랜치 확인
git branch
# * 표시가 있는 브랜치가 현재 위치

# 또는
git status
# 맨 위에 "On branch feature/이름" 표시됨
```
